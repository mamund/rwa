Application-Level Profile Semantics (ALPS)

Abstract

This document describes a media type for defining application-level semantic descriptions (similar to those defined by HTML microformats) in a way that incorporates both human-readable documentation and machine-readable instructions on applying semantic descriptions to elements within a wide range of media-type representations.

 TOC 

1.  Introduction

This document describes a media type for defining application-level semantic descriptions (similar to those defined by HTML microformats) in a way that incorporates both human-readable documentation and machine-readable instructions on applying semantic descriptions to elements within application-agnostic media-type representations (such as HTML, HAL, Collection+JSON, Siren, etc.). This document also identifies an ALPS Registry, sets a standard for creating and registering ALPS document identifers in that registry, and provides suggested conventions for applying ALPS descriptions to existing media types.

This document registers two media-type identifiers with the IANA: 'application/alps+xml' ("ALPS+XML") and 'application/alps+json' ("ALPS+JSON").


 TOC 

1.1.  Notational Conventions

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in [RFC2119] (Bradner, S., “Key words for use in RFCs to Indicate Requirement Levels,” March 1997.).


 TOC 

1.2.  Motivation

When implementing a hypermedia client/server application using a general media type (i.e. HTML, Atom, Collection+JSON, etc.), client and server instances need to share application-specific information such as data element names, link relation values, and state transfer parameters. This information is directly related to the application being implemented (i.e. accounting, contact management, etc.) rather than the media type used in the representations.


 TOC 

1.2.1.  Describing Domain-Specific Semantics

Instead of creating (and registering) an entirely new media type (i.e. 'application/accounting'), representation authors can create an ALPS document that describes a "profile" of the target domain; one that contains the vital appliation-level semantic descriptors and transitions. This profilel can then be consistently applied to a wide range of media types by server implementors and successfully consumed by client applications. This focus on defining appliction-level semantics independent of transfer protocol or media type design makes it possible to increase the specificity of the representations without reducing the generality of the media type casrrying the representation.


 TOC 

1.2.2.  ALPS-based Server Implementations

Server implementors can use ALPS documents as a basis for building domain-specific solutions without having to create their own custom media type or re-invent the vocabulary and transition set for a common domain (e.g. accounting, microblogging, etc.). By using the ALPS document as a guide, servers can map internal data to commonly-understood semantic descriptors and state transitions and increase the likelyhood that existing client applications (those who share the same understanding of the ALPS document) will be able to successfully interact with that server.


 TOC 

1.2.3.  ALPS-based Client Implementations

Armed with an ALPS document as a "profile" for the received representation, client applications can locate and identify semantic descriptors of interest within the response. Client applications can "code for the profile" and better adjust to detailed changes to the response layout.

2.  ALPS Documents

ALPS Documents contain a machine-readable collection of identifying strings and their human-readable explanations. An ALPS document can be represented in either XML or JSON format. This section identifies the general elements and properties of an ALPS document, their meaning, and their use. Additional subsections provide specific details on how to construct a valid ALPS document in XML and in JSON format.


 TOC 

2.1.  Compliance

An implementation is not compliant if it fails to satisfy one or more of the MUST or REQUIRED level requirements. An implementation that satisfies all the MUST or REQUIRED level and all the SHOULD level requirements is said to be "unconditionally compliant"; one that satisfies all the MUST level requirements but not all the SHOULD level requirements is said to be "conditionally compliant."


 TOC 

2.2.  ALPS Document Properties

The ALPS media type defines a small set of properties. These properties appear in both the XML and JSON formats, but are expressed in different ways depending on the format. Below is a list of the properties that can appear in an ALPS document.


 TOC 

2.2.1.  'alps'

Indicates the root of the ALPS document. This property SHOULD have one or more 'element' child properties.

Examples:

XML:
<alps>...</alps>
JSON:
{"alps" : ... }


 TOC 

2.2.2.  'appears'

Indicates the appearance rules for the related 'element' ('element') in a a representation. All valid values for this property are terms defined in [RFC2119] (Bradner, S., “Key words for use in RFCs to Indicate Requirement Levels,” March 1997.), and have the meaning given in that standard.


 TOC 

2.2.2.1.  Applied to a 'top-level' 'element' block

The 'appears' property MAY appear as a property of a "top level" 'element' block. This controls whether the element may appear at the "top level"; that is, not enclosed in any other element defined by this ALPS document. The formal meaning of "top level" varies by the media type; see Section 3 (Instructions on Applying ALPS to Existing Media Types) for definitions.

Examples:

XML:
<element ... appears="MUST NOT">
JSON:
{"element" : "{..., "appears" : "MUST NOT"}}

Valid values for 'appears' in a top-level 'element' block are: "MUST", "SHOULD", "MAY", and "MUST NOT". If this property does not appear, the value SHOULD be treated as "MUST NOT".


 TOC 

2.2.2.2.  Applied to 'non top-level' element block

The 'appears' property MAY appear as a property of a 'non top-level' 'element' block. This controls the rules by which the referenced element may appear as a child of the enclosing 'element'. The formal meaning of "enclosing 'element'" varies by the media type; see Section 3 (Instructions on Applying ALPS to Existing Media Types) for definitions.

Examples:

XML:
<element ... appears="MAY">
JSON:
{"element" : "{..., "appears" : "MAY"}}

Valid values for 'appears' in a 'non top-level' 'element' are: "MUST", "SHOULD", and "MAY". If this property does not appear, the value SHOULD be treated as "MAY".


 TOC 

2.2.3.  'cardinality'

Indicates the number of times the related 'element' ('element') can appear. Valid values are "single" and "many". If this property does not appear, the value SHOULD be treated as "many".

Examples:

XML:
<element ... candinality="many">
JSON:
{"element" : "{..., "cardinaltiy" : "many"}}


 TOC 

2.2.4.  'description'

A text field that contains free-form, usually human-readable, text.

A 'description' block SHOULD appear as a child of 'element' ('element'). When present, it describes the meaning and use of the related 'element'.

XML:
<element ... > <description>...</description> </element>
JSON:
{"element" : {...} {"description" : {"value" : "..."} }

A 'description' block MAY appear as a child of 'alps' ('alps'). When present, it describes the purpose of the ALPS document as a whole.

XML:
<alps> <description>...</description> ... </apls>
JSON:
{"alps : "description" : { "value" : "..."}, ... }

The 'description' element has one OPTIONAL sub-element: 'format' ('format') with two possible values:

  1. "text"
  2. "html"

If the 'format' property is set to "html" the content within the 'description' block MUST be treated as HTML markup. If no 'format' property is present, the content SHOULD be treated as plain text.

XML:
<description format="html"> <h1>Date of Birth</h1> <p>...</p> </description>
JSON:
{"description" : {"format" : "text" : "value" : "Date of Birth ..."} }


 TOC 

2.2.5.  'element'

Identifies a semantic descriptor. One or more 'element' blocks SHOULD appear as children of the 'alps' ('alps') block. It may also appear as a child of itself; the 'element' property can be nested.

The 'element' property MUST have at least an 'id' ('id') or 'href' ('href') property. It MAY have both. Additionally, the 'element' MAY have any of the following proeprties:

  1. 'appears' ('appears')
  2. 'cardinality' ('cardinality')
  3. 'description' ('description')
  4. 'ext' ('ext')
  5. 'name' ('name')
  6. 'type' ('type')

The 'href' property MUST be a dereferencable URL that points to another 'element' block either within the current ALPS document or in another ALPS document.

If 'element' has an 'href' property, then 'element' is inheriting all the attributes and and sub-properties of what is pointed to by 'href'. When 'element' has a property defined locally, that property value takes precedence over any inherited property value. Since there is no limit to the nesting of elements -- even ones linked remotely -- it is important to process all element chains starting from the 'bottom' to make sure you have collected all the available properties and have established the correct value for each of them.


 TOC 

2.2.6.  'ext'

The 'ext' element can be used to extend the ALPS document with author-specific information. It provides a way to customize ALPS documents with additional properties not covered in this specification. This is an OPTIONAL element.

The 'ext' element has two properties. The 'href' ('href') property MUST appear for the 'ext' element. It's value MUST be a URI. The 'value' ('value') property MAY appear for the 'ext' element. The meaning and use of the value property is described in the document found by dereferencing the 'href' property of the 'ext' element.

Examples:

XML:
<ext href="http://alps.io/ext/directions" value="north south east west" >
JSON:
{"ext" : {"href" : "http://alps.io/ext/directions", value="north south east west"}}

The 'ext' element MAY appear as a child of the following elements:

  1. 'alps' ('alps')
  2. 'element' ('element')

Since the 'ext' element has no specific meaning within the this specification, it MUST be ignored by any application that does not understand it's use and meaning.

Authors who define 'ext' elements SHOULD register these extensions with the ALPS Registry (see Section 4 (The ALPS Registry)) and use the URI provided by the registry in the 'name' property of their extension.


 TOC 

2.2.7.  'format'

Indicates whether the associated content in the should be treated as 'text' or 'html'. This property MAY appear as a property of the 'description' ('description') element.


 TOC 

2.2.8.  'href'

Contains a resolvable URI.

When it appears as a property of 'element' ('element'), 'href' points to another 'element' either within the existing ALPS document or in another ALPS document.

When it appears as a property of 'ext' ('ext'), 'href' points to an external document which provides the defintion of the extension.

When it appears as a property of 'link' ('link') 'href' points to an external document whose relationship to the current document is described by the associated 'rel' ('rel') property.


 TOC 

2.2.9.  'id'

A document-wide unique identifier for the related element. This MAY appear as a property of each 'element' ('element').

The value of this property will appear within generic representations as the name of a semantic descriptor or hypermedia control (see Section 3 (Instructions on Applying ALPS to Existing Media Types) for details). The exception is when an element defines 'name' ('name') as well as 'id' ('id'). In that case, the value of 'name' contains the name of the semantic descriptor or hypermedia control, and the value of 'id' is just a unique internal identifier for the current document.

When applied to an ALPS document, a URI fragment identifier points to the element with the appropriate 'id'. For example, the fragment identifier "customer" in the URI http://example.com/#customer refers to an ALPS element with 'id' set to "customer".

A relative URL with a fragment identifier (e.g. "#customer") refers to an element within the ALPS document containing the reference.

The complete URI (including the fragement) forms an "abstract semantic type" identifier. This is a resolvable URI (URL) that can be used as a type indicator within ALPS documents.

In most cases, the 'id' value will be used as the link relation type value for safe transitions. For this reason, when selecting 'id' values and defining 'element' semantics, it is important to avoid creating conflicts with existing IANA-registered values. In cases where this does conflict occur, the results are indeterminate.


 TOC 

2.2.10.  'link'

Identifies a link between the current property and some other external resource. Can be applied to the 'alps' ('alps') and the 'element' ('element') properties.

The 'link' property MUST have two sub-properties: 'href' ('href') and 'rel' ('rel').


 TOC 

2.2.11.  'name'

Indicates the name of the 'element' ('element') as found in generic representations. It MAY appear as a property of 'element'.

This is used when the name of the 'element' or 'child' is used as an 'id' ('id') value elsewhere in the ALPS document. For instance, if a single ALPS document defines an element descriptor called "customer" and a safe-idempotent control called "customer", they can't both have 'id="customer"' in the ALPS document. One of them needs to have some other 'id', and to set 'name="customer"'.


 TOC 

2.2.12.  'rel'

Contains a [RFC5988] (Nottingham, M., “Web Linking,” October 2010.) approved value (e.g. an IANA-registered link relations type or a URI).

Appears as a property of link ('link').


 TOC 

2.2.13.  'type'

Indicates the type of representation control to which the element is applied. This MUST appear for each element. Four valid values are defined:

"descriptor"
A state element (e.g. HTML.SPAN, HTML.INPUT, etc.) OR a hypermedia control that triggers a safe, idempotent transition (i.e. HTTP.GET, HTTP.HEAD, etc.).
"safe"
A hypermedia control that triggers a safe, idempotent state transition (i.e. HTTP.GET, HTTP.HEAD, etc.).
"idempotent"
A hypermedia control that triggers an unsafe, idempotent state transition (i.e. HTTP.PUT, HTTP.DELETE etc.).
"unsafe"
A hypermedia control that triggers an unsafe, non-idempotent state transition (i.e. HTTP.POST).

If no 'type' property is associated with the element then the 'type="descriptor"' is implied.


 TOC 

2.2.14.  'value'

Contains a string value. It MAY appear as a property of the 'description' ('description') and the 'ext' ('ext') elements.


 TOC 

2.2.15.  'version'

Indicates the version number of the ALPS document. This SHOULD appear as a property of the 'alps' ('alps') property. Currently the only valid value is "1.0". If no value appears, then it SHOULD be set to "1.0".


 TOC 

2.3.  ALPS Representations

ALPS documents can be represented in either XML or JSON format. This section contains notes on how the ALPS properties appear in each format along with examples to guide ALPS document authors.


 TOC 

2.3.1.  Sample HTML

Below is a simple HTML document that contains a handful of semantic descriptors and transition instructions. This document was genrated from the XML and JSON ALPS documents that follow. Use this HTML document as a guide when evaluating the XML and JSON examples. 

<!-- sample HTML document -->
<html>
    <head>
        <link rel="profile" href="http://alps.io/documents/search" />
    </head>
    <body>
        <form class="search" action="..." method="get">
            <input type="text" name="search" value="..." />
            <select name="resultType">
                <option value="summary" />
                <option value="detailed" />
            </select>
            <input type="submit" />
        </form>
    </body>
</html>

2.3.2.  XML Representation


 TOC 

2.3.2.1.  XML Representation Notes

In the XML version of ALPS documents, the following properties always appear as XML elements: 'alps' ('alps'), 'description' ('description'), 'element' ('element'), and 'ext' ('ext'). All other ALPS properties appear as XML attributes.


 TOC 

2.3.2.2.  Complete XML Representation

Below is an example of an applcation/alps+xml representation. 

<?xml version="1.0"?>
<alps version="1.0">
    <description format="html">
        <h1>A Sample ALPS+XML Document</h1>
    </description>

    <element id="search"
        type="safe"
        cardinality="single"
        appears="MUST NOT">
        <description format="text">
            A search form with a two inputs.
        </description>
        <element href="#resultType" />
        <element id="value"
            name="search"
            type="descriptor"
            cardinality="single"
            appears="MAY">
            <description>input for search</description>
        </element>
    </element>

    <element id="resultType"
        type="descriptor"
        cardinality="single"
        appears="MAY"
        <description>results format</description>
        <ext
            href="http://alps.io/ext/range"
            value="summary,detail" />
    </element>
</alps>

2.3.3.1.  JSON Representation Notes

When representing ALPS documents in the JSON format, the 'element' ('element') and 'ext' ('ext') properties are always express as arrays of anonymous objects - even when there is only one member in the array.

For example: 

"element" : [
    {
        "id" : "search",
        "type" : "safe"
        "element" : [
            {"href" : "#value"},
            {
                "id" : "resultType",
                "type" : "descriptor",
                "ext" : [
                    {
                        "href" : "http://alps.io/ext/range",
                        "value" : "summary,detail"
                    }
                ]
            }
        ]
    },
    {
        "id" : "#value",
        "type" : "descriptor"
    }

]
 Arrays in ALPS+JSON 

The 'description' ('description') description is always expressed as a named object.


For example: 

{
    "description" : {
        "format" : "text",
        "value" : "Rules are imporant"
    }

3.4.  Linking to ALPS Documents

When a representation links to an ALPS document describing its semantics, the "profile" link relation (found in the IANA link registry) SHOULD be used to describe the relationship between the representation and the ALPS document.

A single representation may be "profile"-ed more than one ALPS document. If two ALPS documents give conflicting semantics for the same element, the document linked to earlier in the representation SHOULD take precedence.
